🔧 Engine Architecture

  • Minimax with Alpha-Beta Pruning — the foundational search algorithm. Explores a game tree to find the best move, pruning branches that cannot possibly affect the result. Reduces nodes searched from O(b^d) to O(b^(d/2)).
  • Quiescence Search — extends the search at leaf nodes to only examine captures, preventing the "horizon effect" where the engine stops just before a significant exchange.
  • Transposition Table — a hash map (keyed by FEN position) that caches evaluated positions. When the same position is reached via different move orders, the cached result is reused instantly. Capped at 80,000 entries with LRU eviction.
  • MVV-LVA Move Ordering — sorts moves before searching: captures of high-value pieces by low-value pieces first, then promotions, then checks. Better ordering means more alpha-beta cutoffs.
  • Piece-Square Tables — 64-value arrays for all 6 piece types encoding positional bonuses. Knights prefer the center; kings prefer safety; rooks prefer open files; pawns prefer advancing.
  • Iterative Deepening — the engine searches depth 1 first, then 2, up to maxDepth, so it always has a valid best move even if time runs out (not applicable in single-threaded JS but architecturally sound).

📚 Libraries Used

Constants & Piece Values

The material worth of each chess piece in centipawns (1 pawn = 100). These values are industry-standard and match those used by most competitive engines.

constants.js 
const PV = {
  p: 100,   // pawn   — baseline unit
  n: 320,   // knight — slightly less than bishop
  b: 330,   // bishop — slightly more than knight
  r: 500,   // rook   — 5 pawns
  q: 900,   // queen  — 9 pawns
  k: 20000  // king   — effectively infinite
};
Why these numbers?

Centipawn values come from decades of engine tuning. Bishop > Knight by 10cp because bishops control longer diagonals in open positions. The king's value (20000) must exceed the total material on the board (~10000cp) so the engine never trades the king for any material gain.

Piece-Square Tables

64-element arrays encoding a bonus (or penalty) for each piece on each square. This gives the engine a positional understanding beyond raw material counting.

evaluation.js 
const PST = {
  p: [ // PAWN — read from White's perspective (rank 8 → rank 1)
     0,  0,  0,  0,  0,  0,  0,  0,  // rank 8
    50, 50, 50, 50, 50, 50, 50, 50,  // rank 7 — near promotion
    10, 10, 20, 30, 30, 20, 10, 10,  // rank 6 — center advance bonus
     5,  5, 10, 25, 25, 10,  5,  5,  // rank 5
     0,  0,  0, 20, 20,  0,  0,  0,  // rank 4 — d/e pawn push
     5, -5,-10,  0,  0,-10, -5,  5,  // rank 3 — penalise blocking d/e
     5, 10, 10,-20,-20, 10, 10,  5,  // rank 2 — starting rank
     0,  0,  0,  0,  0,  0,  0,  0   // rank 1 — impossible for pawn
  ],
  // knight, bishop, rook, queen, king tables follow same pattern …
};
How it works

For each piece on the board, we look up PST[piece.type][squareIndex] and add it to the score (positive for White, negative for Black).

The index is computed as (7-rank)*8 + file for White pieces (flipping the board) so the table always reads from White's perspective.

  • Knights get huge bonuses in the center
  • Rooks get bonuses on the 7th rank
  • Kings get penalties in the center (endgame table differs)
⚖️

Static Evaluation Function

Assigns a numeric score to any board position. Positive = good for White, Negative = good for Black. Called at leaf nodes of the search tree.

evaluation.js 
function evaluate(g) {
  // Terminal states — checked before material count
  if (g.in_checkmate())
    return g.turn() === 'w' ? -99999 : 99999;
  if (g.in_draw() || g.in_stalemate() ||
      g.in_threefold_repetition())
    return 0;

  let score = 0;
  const bd = g.board();  // 8×8 array of piece objects

  for (let r = 0; r < 8; r++) {
    for (let c = 0; c < 8; c++) {
      const p = bd[r][c];
      if (!p) continue;

      const base = PV[p.type];
      const idx  = p.color === 'w'
        ? (7 - r) * 8 + c   // flip for white
        : r * 8 + c;          // as-is for black
      const pst  = PST[p.type][idx];

      score += p.color === 'w'
        ? (base + pst)
        : -(base + pst);
    }
  }
  return score;
}
Score interpretation

+100 = White is up a pawn.
+500 = White is up a rook.
-900 = Black is up a queen.
±99999 = checkmate.

The PST bonus can be up to ~50cp per piece, so positional play matters as much as being up a pawn.

Transposition Table

A hash map from board position (FEN string) to previously computed evaluation scores. Prevents re-evaluating the same position reached via different move orders.

transposition.js 
const TT     = new Map();
const TT_MAX = 80_000;  // evict oldest at this size

function ttGet(fen, depth) {
  const e = TT.get(fen);
  // Only use cached result if it was searched at least
  // as deep as the current requested depth
  return (e && e.d >= depth) ? e.v : null;
}

function ttSet(fen, depth, val) {
  // LRU eviction: delete the oldest inserted key
  if (TT.size >= TT_MAX)
    TT.delete(TT.keys().next().value);
  TT.set(fen, { d: depth, v: val });
}
Why it matters

In chess, transpositions — reaching the same position via different move orders — are extremely common. Without a TT, the engine redundantly re-evaluates identical positions.

With the TT, depth-5 search effectively performs like depth-6 or 7 in practice because entire sub-trees are skipped.

FEN (Forsyth-Edwards Notation) encodes full board state including castling rights and en passant — making it a perfect unique key.

🗂

Move Ordering — MVV-LVA

Alpha-beta pruning is most effective when the best moves are searched first. MVV-LVA (Most Valuable Victim — Least Valuable Attacker) scores moves before searching them.

engine.js 
function mvvlva(m) {
  let s = 0;

  // Captures: 10× victim value − attacker value
  // QxP scores higher than PxQ (risky)
  if (m.captured)
    s += 10 * (PV[m.captured] || 0) - (PV[m.piece] || 0);

  // Promotion: always a high-priority move
  if (m.flags && m.flags.includes('p')) s += 800;

  // Checks: applying pressure is usually good
  if (m.san && m.san.includes('+'))    s +=  50;

  return s;
}

function orderMoves(moves) {
  return moves.sort((a, b) => mvvlva(b) - mvvlva(a));
}
Why ordering is critical

Alpha-beta pruning in the worst case (random ordering) examines O(b^d) nodes — the same as plain Minimax.

With perfect ordering it examines O(b^(d/2)) — effectively doubling search depth for free.

MVV-LVA gets us close to perfect ordering for tactical positions where captures dominate.

Quiescence Search

Extends the search at leaf nodes to resolve all capture sequences before calling the static evaluator. Prevents the "horizon effect".

quiescence.js 
function qSearch(g, alpha, beta, ply) {
  // Stand-pat: the current position's static score.
  // We can always "do nothing" instead of capturing.
  const stand = evaluate(g);

  if (stand >= beta)  return beta;   // beta cutoff
  if (stand > alpha) alpha = stand;  // raise alpha floor
  if (ply > 5)       return alpha;   // depth safety cap

  // Only examine capture moves (not quiet moves)
  const caps = orderMoves(
    g.moves({ verbose: true })
     .filter(m => m.captured)
  );

  for (const m of caps) {
    g.move(m);
    const sc = -qSearch(g, -beta, -alpha, ply + 1);
    g.undo();
    if (sc >= beta)  return beta;
    if (sc > alpha) alpha = sc;
  }
  return alpha;
}
The horizon effect

Without qSearch, the engine might evaluate a position where a queen is about to be captured as "+900cp" because it can't see one move further. The engine would make catastrophically bad moves.

Stand-pat is the key insight: if the current position is already so good we'd accept it, we don't need to examine captures — we return alpha immediately.

The ply cap of 5 prevents infinite loops in endless exchange chains.

🌳

Alpha-Beta Minimax

The core search algorithm. Recursively explores the game tree, keeping track of the best scores each side can guarantee. Prunes branches that cannot affect the result.

minimax.js 
function minimax(g, depth, alpha, beta, maximising) {
  const fen = g.fen();

  // ① Transposition table lookup
  const hit = ttGet(fen, depth);
  if (hit !== null) return hit;

  // ② Leaf node → quiescence search (not static eval)
  if (depth === 0) {
    const v = qSearch(g, alpha, beta, 0);
    ttSet(fen, 0, v); return v;
  }

  // ③ Terminal position checks
  if (g.in_checkmate()) {
    const v = maximising ? -99999 : 99999;
    ttSet(fen, depth, v); return v;
  }
  if (g.in_draw() || g.in_stalemate()) {
    ttSet(fen, depth, 0); return 0;
  }

  // ④ Ordered move iteration
  const moves = orderMoves(g.moves({ verbose: true }));
  let best = maximising ? -Infinity : Infinity;

  for (const m of moves) {
    g.move(m);
    const val = minimax(g, depth-1, alpha, beta, !maximising);
    g.undo();

    if (maximising) {
      best  = Math.max(best, val);
      alpha = Math.max(alpha, val);
    } else {
      best = Math.min(best, val);
      beta = Math.min(beta, val);
    }
    // ⑤ Alpha-beta cutoff (prune this branch)
    if (beta <= alpha) break;
  }

  ttSet(fen, depth, best);
  return best;
}
Alpha-Beta explained

alpha = best score the maximising player (White) can guarantee so far.
beta = best score the minimising player (Black) can guarantee so far.

When beta ≤ alpha, the current branch is pruned: the opponent already has a better option elsewhere, so they'd never let us reach this position.

  • Step ① — reuse cached result if available
  • Step ② — at depth 0, resolve captures before evaluating
  • Step ③ — checkmate/draw are immediate returns
  • Step ④ — search best moves first (from MVV-LVA)
  • Step ⑤ — prune if this branch is irrelevant

Difficulty System

Six levels with distinct characters. Lower levels introduce controlled randomness to simulate human error. Higher levels are pure engine play at increasing depths.

engine.js 
const LEVELS = {
  novice:       { depth: 1, rand: 0.70 },  // 70% random
  beginner:     { depth: 2, rand: 0.40 },  // 40% random
  intermediate: { depth: 3, rand: 0.12 },  // 12% random
  expert:       { depth: 3, rand: 0.00 },  // pure engine
  master:       { depth: 4, rand: 0.00 },  // deeper search
  grandmaster:  { depth: 5, rand: 0.00 },  // elite depth
};

function getBestMove(g, lvl) {
  const cfg   = LEVELS[lvl] || LEVELS.intermediate;
  const moves = g.moves({ verbose: true });
  if (!moves.length) return null;

  // Randomness injection for lower levels
  if (cfg.rand > 0 && Math.random() < cfg.rand)
    return moves[Math.floor(Math.random() * moves.length)];

  // Engine search for all other cases
  const isW = g.turn() === 'w';
  let best = null, bestVal = isW ? -Infinity : Infinity;

  for (const m of orderMoves([...moves])) {
    g.move(m);
    const val = minimax(g, cfg.depth-1,
                        -Infinity, Infinity, !isW);
    g.undo();
    if (isW ? val > bestVal : val < bestVal)
      { bestVal = val; best = m; }
  }
  return best || moves[0];
}
Design decision

Pure depth reduction (depth 1 vs 5) alone doesn't create a believable novice. A depth-1 engine still plays tactically sound, just short-sighted.

Random injection creates realistic human-like blunders — missing a hanging piece, misplacing a knight — that make the game feel like a real beginner opponent.

Hints always use Expert-level evaluation (depth 3, no randomness) regardless of the current game difficulty.

UI & Game Loop

Connects the chess engine to the chessboard.js UI. Handles drag-drop moves, square highlights, move history, captured pieces, and the hint system.

controller.js 
// ── Game Loop ──────────────────────────────────
function onDrop(src, tgt) {
  if (over || g.turn() !== 'w') return 'snapback';
  clearHL();

  // Validate and apply the human's move
  const mv = g.move({ from: src, to: tgt, promotion: 'q' });
  if (!mv) return 'snapback';  // illegal move

  markLastMove(); markCheck();
  refreshStatus(); refreshMoves(); refreshCaptured();

  // Trigger engine response on next tick
  if (!g.game_over()) engineMove();
  else { refreshStatus(); endGame(); }
}

// ── Hint System ────────────────────────────────
function doHint() {
  if (hints <= 0 || over || g.turn() !== 'w') return;
  hints--; refreshHints(); clearHL();

  // Always use expert evaluation for hints
  const hint = getBestMove(g, 'expert');
  if (!hint) return;

  // Highlight from-square (light green) and to-square (bright green)
  hlSq(hint.from, 'sq-hint-from');
  hlSq(hint.to,   'sq-hint-to');
  setStatus(`Hint: Consider ${hint.from} → ${hint.to} (${hint.san})`);

  // Auto-clear hint highlight after 3.5 seconds
  hintTimer = setTimeout(() => {
    clearHL(); markLastMove(); refreshStatus();
  }, 3500);
}
Architecture

The game loop is event-driven: the human drops a piece → onDrop fires → engine response is deferred with setTimeout(60ms) to let the browser repaint first.

All UI state is derived from the chess.js game object g — never stored separately. This means the UI can never get out of sync with the game state.

The hint system consumes from a fixed pool of 6 per game. Each hint costs one pip, and the button disables when the pool is empty or it's the engine's turn.
